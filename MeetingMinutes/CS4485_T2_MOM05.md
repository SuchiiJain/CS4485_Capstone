## Discussion Summary

**1. Action Item Follow-Ups from MOM #4**
All action items from MOM #4 were reviewed. Test repositories still need to be pushed to GitHub to validate the CI pipeline. FastAPI backend work has begun on a dedicated branch.

**2. Frontend Progress**
Updated the frontend styling to match Figma wireframes and added additional wireframe screens covering most essential views including Dashboard, Projects, Issues, History, and Config. Added static screens for Login, Empty State, Issue Drawer, and User Settings on a separate branch. The team agreed to divide remaining wireframe screens among available members via Jira.

**3. FastAPI Backend & Database Integration**
Created a new FastAPI branch on the backend repo currently reading from existing JSON files as an interim step before full database integration. Team members are actively working on the database schema. The team noted the database still needs tables for scan reports and issues, and needs to be hooked up to the pipeline. Database work should be done on a branch rather than directly to main.

**4. VS Code Extension Discussion**
A VS Code Extension was discussed as a future improvement — instead of manually adding `.docrot-config.json` to each repo, the extension would handle configuration automatically for all users upon download. Portia plans to look into this after midterms.

**5. In-Meeting Discussion**
The team discussed several ideas for improving usability and development coordination. One proposed improvement was adding a configuration **wizard** within the VS Code extension to help users map the `.docrot-config.json` file to their code repositories more easily. This would reduce manual setup and make the system more user-friendly.
The team also discussed the importance of **updating and maintaining the project specification file** so that it accurately reflects the current system architecture and development progress. Keeping the spec file updated would help the team remain aligned even when members are working asynchronously or unable to meet regularly. For example, if the specification is shared with external tools or AI assistants, it could help ensure that generated work remains consistent with the current project design.
Another suggestion was to **store the specification file directly in the repository**, allowing tools or contributors to reference it before making changes. This would allow instructions such as “read the spec file before writing any code,” ensuring development follows the intended system design.
The team also discussed making the system more **user-friendly** so that users do not have to manually perform tasks such as managing documentation updates.
Additionally, the **Figma wireframes should be updated** to better represent a web-based dashboard layout rather than a mobile-style interface.
Finally, the team noted several broader system design considerations mentioned by the professor, including the need to think about **security layers, database optimization, and potential caching strategies** as the project architecture evolves.
