### 9. In-Meeting Discussion

**Scalability Improvements (prompted by professor feedback):**

The professor raised questions about scalability. The team presented several improvements made this sprint:

- **Repo-stamped issue tracking:** Closing an issue now tells Firebase which repo ID it belongs to, so issues are correctly read and resolved per repo.
- **Parallel query execution:** Previously, only one repository was scanned at a time. Queries now run concurrently, significantly reducing load times.
- **Faster rendering:** Overall rendering performance has been improved.
- **Security fix (in progress):** A bug was identified where a user could potentially view another user's repository. A fix has been implemented and is currently being confirmed.
- **Change detection:** A change detection feature has been implemented to track documentation drift over time.
- **Auto-fix functionality:** An apply/fix function for repositories has been added.

**Professor's Question — Data Storage and Scalability:**

The professor asked: *"If you have a business, what do you store in your database for your codebase? Do you store anything at all, or do you just check and authenticate user accounts and then run? Is anything stored in the database?"*

The team responded that Firebase does store data from scans. The professor suggested that minimizing what is stored in the database would improve scalability. The team clarified that they do not store entire codebases — only relevant flagged content — but acknowledged they do not have a quantifiable metric for how much is stored. The broader question of what happens if a codebase is deleted (i.e., whether stored data becomes stale or orphaned) was noted as something to think through further.

**Other Open Issues:**

- AI suggestions integration is still unstable; the team suspects the issue may be on Groq's end and will continue monitoring.
- Portia is able to view all users' scans, likely due to having admin privileges on Firebase. This needs to be resolved before final demo.
