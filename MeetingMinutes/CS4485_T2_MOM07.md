### 8. In-Meeting Discussion
 
**Pipeline Overview:**
The team walked through the current end-to-end flow: users add the `.docrot.yml` workflow and config file to their repository, which triggers the GitHub Action on every push. The action runs the scanner on GitHub's infrastructure and sends fingerprint scan results to the backend hosted on Google Cloud. GitHub OIDC and Workload Identity Federation authenticate the pipeline securely without static secrets. Scan results are written to Firestore and surfaced on the frontend dashboard. The scanner does not run locally — all scanning happens on GitHub's side. The team noted that by next week, user-specific scans and flags should be fully visible and linked on the dashboard.
 
**Fingerprint Comparison Logic:**
The TA raised a question about which fingerprints are being compared when multiple contributors push to the same branch. The team clarified that whichever push arrives first establishes the stored fingerprint baseline. Each subsequent push compares against the most recently stored fingerprint, not the original. The system tracks incremental changes function-by-function across every push. Minor changes such as variable renames that do not cross a scoring threshold will not trigger a documentation flag. The TA noted that a log of all comparisons should be maintained. The team confirmed this is the intent — each push event is logged, flags are stored (not overwritten by new pushes), and all history is viewable on the dashboard. A GitHub issue is also created per flag so contributors have direct visibility in their repository.
 
**Scalability Discussion:**
The professor raised scalability as a key consideration, asking how the system would handle high volumes of concurrent requests — for example, 10,000 pushes arriving simultaneously. Key points discussed:
 
- **Input queue:** Incoming requests to the backend should be queued to prevent data loss under heavy load. Google Cloud Functions handle some concurrency natively, but an explicit queue mechanism may be necessary at scale.
- **Worker threads:** Parallel processing of scan jobs could reduce bottlenecks and improve throughput.
- **Google Cloud as a potential bottleneck:** The database and cloud function layer may become a constraint under heavy traffic and would need profiling and optimization.
- **Response time expectations:** Documentation rot detection is not always time-critical — a delay of one to five minutes is generally acceptable. However, if a user explicitly requests a scan, responsiveness matters more. The system should not drop requests even if processing is delayed.
- **Agent-driven code changes:** The professor noted that AI coding agents operate at speeds far exceeding human contributors, meaning the volume of pushes and frequency of documentation mismatches could increase significantly in future use cases. The system design should account for this.
 
**Documentation Scope:**
The TA clarified that the system should only check the latest version of documentation — not maintain or compare historical documentation states independently. Each scan should evaluate whether the current documentation reflects the current code at the time of the push.
 
**Commercial Viability:**
The professor encouraged the team to think about how this tool could be generalized beyond documentation rot detection. The core pipeline — a GitHub Action that scans code, sends structured results to a cloud backend, and surfaces insights in a dashboard — is applicable to many categories of code quality tooling. The professor noted the system already incorporates continuous integration principles and encouraged the team to consider what would make it commercially viable, including scalability, accessibility, and extensibility.
